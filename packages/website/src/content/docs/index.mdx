## One platform, every methodology

Most P3M tools force a choice. TruePPM is designed from the ground up so teams can work the way they work — and every stakeholder still sees what they need.

  </Card>

## 5-minute local demo

:::tip[Just want to look, not install?]
Two zero-config paths — a hosted read-only demo at `try.trueppm.dev` and a
one-command local demo (`docker compose -f docker-compose.demo.yml up`) — **ship
with the 0.4 beta**; the [**Try it**](/getting-started/try-it/) page previews
them. Until 0.4 tags, the dev stack below is the fastest way to a populated
instance — six persona logins, full write access.
:::

```bash
git clone https://gitlab.com/trueppm/trueppm.git && cd trueppm
docker compose up -d
docker compose exec api python manage.py seed_demo_project --with-personas
# Web UI → http://localhost:5173   (six persona logins, password: demo)
# API    → http://localhost:8000/api/schema/swagger-ui/
```

The persona password is `demo` because the local Docker stack runs with `DEBUG=True`; on a non-dev install the seed prints a one-time random password unless you set `TRUEPPM_DEMO_PASSWORD`. See [`seed_demo_project`](/administration/management-commands/#seed_demo_project).

Or use the scheduling engine standalone — no server required:

```bash
pip install trueppm-scheduler
```

[View `trueppm-scheduler` on PyPI →](https://pypi.org/project/trueppm-scheduler/)

```python
from datetime import date, timedelta
from trueppm_scheduler import schedule, Calendar, Project, Task, Dependency

calendar = Calendar()  # Mon–Fri working days
project = Project(
    id="p-1", name="My Project", start_date=date(2026, 1, 5),
    tasks=[
        Task(id="t-1", name="Design", duration=timedelta(days=5)),
        Task(id="t-2", name="Build",  duration=timedelta(days=10)),
    ],
    dependencies=[Dependency(predecessor_id="t-1", successor_id="t-2")],
    calendar=calendar,
)
result = schedule(project)
print(result.tasks[1].early_finish)   # 2026-01-23
print(result.critical_path)           # ['t-1', 't-2']
```

**Stack:** Django 5.2 + Channels · React 19 + Vite · PostgreSQL 16 + ltree · Apache 2.0

## Why TruePPM?

<CardGrid>
  <Card title="Real scheduling, not task lists" icon="rocket">
    CPM forward/backward pass on every change. You always know which tasks drive your deadline and where you have float. All 4 dependency types with calendar-aware lag.
  </Card>
  <Card title="Monte Carlo risk analysis" icon="random">
    Add three-point estimates and get P50/P80/P95 completion dates. Commit to P80 — there's only a 50% chance you'll hit the CPM date.
  </Card>
  <Card title="Computed, not guessed" icon="puzzle">
    Every date and forecast is calculated by the engine, with the derivation to show for it. The read-only AI-query server (MCP) landing in the 0.4 beta answers from that engine — an AI client can *ask* your schedule, but a language model never *invents* your dates. [The story →](/the-story/)
  </Card>
  <Card title="Self-hosted, Apache 2.0" icon="open-book">
    Your data stays yours. Run it on your infrastructure. The scheduling engine ships standalone on PyPI — use it without the API if you just need the math.
  </Card>
  <Card title="Built for collaboration" icon="laptop">
    Real-time WebSocket push, 5-role RBAC, offline-first sync for unreliable networks. Every mutation broadcast to connected clients.
  </Card>
</CardGrid>

## What's built today

The **Since** column is the release a capability first shipped in. 0.3 (the `0.3.0-alpha.1` pre-release) is the current build; the 0.1 and 0.2 rows are the foundation it builds on.

| Capability | Since | Description |
|-----------|--------|-------------|
| Scheduling engine | **0.1** | CPM + Monte Carlo. Ships standalone as [`trueppm-scheduler`](https://pypi.org/project/trueppm-scheduler/) on PyPI. |
| REST API | **0.1** | Full CRUD for projects, tasks, dependencies, resources, calendars, members, sprints. OpenAPI 3.0.3 schema. |
| RBAC | **0.1** | 5-role per-project permissions (Owner / Admin / Scheduler / Member / Viewer) on every endpoint. |
| Real-time | **0.1** | WebSocket broadcasts for all mutations, deferred to transaction commit. |
| Offline sync | **0.1** | WatermelonDB-compatible delta protocol with soft-delete tombstones. |
| Schedule (Gantt) | **0.1** | Custom canvas renderer — critical path, baseline comparison, milestones, build mode. |
| Board / Kanban | **0.1** | 5-column board with WIP limit overload detection and drag-to-status. |
| Sprints | **0.1** | Full plan → activate → close lifecycle, burndown, velocity, capacity preflight, retrospective. |
| Risk Register | **0.1** | Probability × impact scoring, lifecycle states, task links, CSV export. |

| Recurring tasks | **0.2** | Recurrence-rule generator with custom and fractional work hours, excluded from the CPM critical path. |
| Durable execution | **0.2** | Outbox-dispatch hardening, Beat heartbeat, dead-letter alerting, retention purge editor, and Idempotency-Key support. |

| Notifications | **0.2** | Per-user email and Slack-webhook notifications with a preference matrix; Git-aware task automation. |
| MS Project import UI | **0.2** | In-app create-from-import flow layered on the 0.1 import / export REST endpoints. |
| Sample projects | **0.3** | Four richly-modeled sample projects (waterfall, agile, hybrid, and a multi-project program) loadable in one click, plus universal JSON project import / export. |
