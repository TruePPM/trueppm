Design system v1.0

TruePPM UI/UX specification

Design system, screen specifications, and interaction patterns

Kelly Hair · TruePPM, Inc. · trueppm.com · March 2026

This document defines the visual language, component library, screen layouts, and interaction patterns for TruePPM across web and mobile platforms. It serves as the single source of truth for all frontend development.

1. Brand identity

1.1 Logo usage

The TruePPM wordmark uses a custom weight of the primary typeface. "True" is regular weight, "PPM" is semibold. Primary color is Blue 600 (#185FA5). On dark backgrounds, use white. Minimum clear space equals the height of the "P" character on all sides.

Logo specifications

  • Primary: "TruePPM" in Blue 600 on white/light backgrounds
  • Inverse: "TruePPM" in white on dark/colored backgrounds
  • Monochrome: "TruePPM" in Text Primary for in-app chrome
  • Minimum size: 80px wide (web), 60px wide (mobile)
  • Never stretch, rotate, apply effects, or change the color split

1.2 Color system

TruePPM uses a semantic color system where colors encode meaning, not decoration. Every color has a specific purpose. The palette is designed for WCAG 2.1 AA contrast compliance in both light and dark modes.

1.5 Borders & surfaces

Design principles

  • No drop shadows. Depth is communicated through surface color (bg-primary on bg-secondary) and borders, never shadows. This keeps the interface flat and crisp at all zoom levels.
  • 0.5px borders on cards, panels, and dividers. 1px on input fields. 2px only on focus rings and the "featured" card accent.
  • Surfaces: bg-primary (white) for content, bg-secondary (#f8f7f5) for chrome/toolbars, bg-tertiary (#f1efe8) for page background.
  • Transparency: use rgba overlays for hover states (rgba(0,0,0,0.04) on light, rgba(255,255,255,0.06) on dark), never solid color changes.

The primary screen for project managers. Split pane: task list (280px fixed) + timeline (remaining width, horizontally scrollable). The task list and timeline scroll vertically in sync.

Timeline panel

Gantt bar types

  • Normal task: 18px tall, Blue 400 fill, 3px border-radius. Progress shown as lighter inner bar from left. Label (task name) inside bar if bar is wide enough (>80px), otherwise to the right.
  • Critical path task: Same dimensions as normal, Red 400 fill. Indicates zero total float.
  • Complete task: Green 400 fill, 100% progress overlay.
  • Summary task: 8px tall, Gray 400, positioned at vertical center of row. Spans from earliest child start to latest child finish. Brackets (downward ticks) at each end.
  • Milestone: 12px diamond (rotated square), Amber 400. No width (zero duration).
  • Baseline ghost: When baseline overlay is active, original planned dates shown as 6px tall semi-transparent bar below the current bar. Gray 200 fill, 40% opacity.

Live impact simulation (killer feature)

  • On drag START of any task bar, the WASM CPM engine runs incremental scheduling from that task downstream.
  • Affected downstream tasks show "ghost" bars at their new positions (Blue 100 fill, dashed outline).
  • A tooltip appears: "Milestone 4 moves from Apr 15 to Apr 18 (+3 days)" near the most impacted milestone.
  • If the change causes a task to become critical (float drops to 0), its ghost bar renders in Red 100 with a flash animation.
  • On drag END, the change commits to the API. The ghost bars animate to their final positions (200ms ease-out) and become solid bars.
  • On drag CANCEL (Escape or release at original position), ghosts disappear immediately.
  • Target: entire ghost rendering pipeline runs in <10ms per frame, maintaining 60fps during drag.

Monte Carlo confidence display

  • Below the last task row, a special row shows the Monte Carlo probability bars.
  • P50 bar: Green 200 fill with Green 400 border, solid. Label "P50 Jun 15" inside.
  • P80 bar: Amber 100 fill with Amber 400 border, dashed. Label "P80 Jul 2" inside.
  • P95 bar (if enabled): Red 50 fill with Red 200 border, dotted. Shown only if user enables "full distribution" mode.
  • The P80 date also appears in the top bar badge for constant visibility.
  • On hover over the MC row, a tooltip shows the full distribution curve (mini histogram).

2.3 Board view (Kanban)

Kanban board with configurable columns (default: To Do, In Progress, Review, Done). Cards show task name, assignee avatar, due date, and progress bar. Drag cards between columns to update status.

Visiban bridge integration

  • If Visiban is connected, the Board view embeds the Visiban board directly (via iframe or API).
  • Moving a card in Visiban updates task status and percent_complete in TruePPM.
  • Dwell time data from Visiban feeds into the Monte Carlo engine (actual cycle times improve future estimates).
  • Visual indicator: small Visiban logo on the Board tab when bridge is active.

2.4 Resource view

Horizontal timeline per resource (rows = people, columns = time). Each cell shows task assignments as colored blocks. Overallocation highlighted with red background. Click an overallocated cell to see conflicting tasks and resolve.

4.2 My Tasks screen

Layout

  • Header: "My tasks" title (17px/600) + subtitle "N due this week" (12px, text-secondary).
  • Section labels: "Overdue", "Due this week", "Upcoming" (11px, uppercase, text-secondary).
  • Task cards: 0.5px border, 10px radius. Overdue cards get a 3px left border in Red 400.
  • Card content: task name (14px/500), metadata row (badges + date, 11px), progress bar (4px tall).
  • Badges: "Critical path" (red-bg/red-text), "Due Apr 18" (amber-bg/amber-text), "On track" (green-bg/green-text).
  • Interactions: tap card to open detail, swipe right to mark complete (green background reveal), long-press to drag-reorder.

4.3 Time entry screen (killer feature)

Layout

  • Prompt: "What did you work on today?" (15px/500) at top.
  • Hint: arrow icons + "Swipe right to log, swipe left to skip" (10px, text-tertiary).
  • Task rows: each shows task name (13px/500), project name (11px, text-secondary), calendar suggestion if available (10px, text-tertiary, e.g., "Calendar: 2pm-4pm design review").
  • Hours input: 48x36px rounded rectangle with pre-filled hours from calendar inference. Tap to edit with number pad. "---" if no suggestion.
  • Submit button: blue-bg rounded rect, centered, "Submit N.N hours" (12px/500).
  • Swipe gestures: right = log hours (green reveal), left = skip/dismiss (gray reveal).
! The 15-second time entry target is the north star metric for this screen. Every design decision should be evaluated against: "does this make time entry faster or slower?" If the user has to tap more than 3 times to log a day's work, we've failed.

4.4 Projects screen

Layout

  • Header: "Projects" title.
  • Offline banner (when disconnected): amber background, "Working offline --- N changes pending sync" (11px/500).
  • Project cards: 0.5px border, 10px radius. Header row: project name (14px/500) + health score circle (36px, color-coded).
  • Stats row: 3-column grid showing Completion % (15px/500 value + 11px label), At-risk count, P80 date.
  • Tap card: navigate to project detail (task list + simplified timeline on tablet).

8. Animation & motion

8.1 Timing functions

8.2 Motion principles

  • Causality: Motion should show cause and effect. A task bar that moves downstream because of a dependency change should animate FROM its old position, not just appear at the new one.
  • Restraint: Only animate things that help the user understand what happened. Schedule recalculation: animate. Menu opening: just appear (100ms fade max).
  • Accessibility: All animations wrapped in @media (prefers-reduced-motion: no-preference). With reduced motion, all transitions are instant.
  • Performance: Only animate transform and opacity (GPU-composited properties). Never animate width, height, top, or left during the Gantt drag interaction.

9. Dark mode

TruePPM supports automatic dark mode via prefers-color-scheme: dark media query. All colors use CSS custom properties that switch between light and dark values. Gantt bars, badges, and health score circles use the same semantic meaning in both modes (red = critical in both).

Dark mode rules

  • Surfaces invert: bg-primary becomes near-black (#1a1a19), bg-secondary becomes dark gray (#2c2c2a).
  • Text inverts: text-primary becomes near-white (#e8e6de).
  • Borders: rgba(255,255,255,0.1) instead of rgba(0,0,0,0.1).
  • Gantt bars: same hues but slightly lighter (400 → 300 stop equivalent) for visibility on dark backgrounds.
  • Health score circles: use 800 fill + 200 text (inverted from light mode 50 fill + 800 text).
  • Badges: same rule — dark fills with light text in dark mode.
  • Never use hardcoded colors. Always go through CSS variables.

10. Implementation notes for developers

10.1 CSS architecture

  • Use Tailwind CSS for web (packages/web). Custom theme extending Tailwind with TruePPM design tokens.
  • Use NativeWind for mobile (packages/mobile). Same Tailwind classes, compiled to React Native StyleSheet.
  • All design tokens defined in tailwind.config.ts as a single source of truth.
  • Gantt chart: custom CSS (not Tailwind) for the SVAR React Gantt integration. Keep in a dedicated gantt.css file that overrides SVAR defaults to match TruePPM design system.
  • No CSS-in-JS runtime (no styled-components, no emotion). Tailwind utility classes only.

10.2 Component file structure

  • One component per file. Named exports only (no default exports).
  • Co-locate test file: TaskCard.tsx + TaskCard.test.tsx
  • Shared components in packages/web/src/components/
  • Feature-specific components in packages/web/src/features/{feature}/components/
  • Mobile components mirror web structure where possible for concept parity.

10.3 Accessibility checklist for every component

  • Interactive elements: role, aria-label or aria-labelledby, tabIndex
  • Dynamic content: aria-live="polite" for updates (schedule recalculated, sync complete)
  • Color: never the only indicator. Add icon, pattern, or text alongside color coding.
  • Focus: visible focus ring on every interactive element. Logical tab order.
  • Gantt bars: each bar is a focusable element with aria-label describing task name, dates, and status.
  • Screen reader: announce "Critical path" and "Overdue" status when task is focused.
